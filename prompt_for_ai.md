# Задача: улучшить региональный прогноз погоды на базе GNN

## Контекст проекта

Мы разрабатываем **GraphCast-lite** — GNN-модель прогноза погоды, архитектурно близкую к Google DeepMind GraphCast. Модель обучена на данных ERA5 реанализа.

Архитектура:
- **Encoder** (MLP + GCN): проецирует grid-ноды → icosahedral mesh
- **Processor** (InteractionNet, 12 шагов message passing): обрабатывает на mesh
- **Decoder** (MLP + GCN): проецирует mesh → grid

Grid: 512×256 (~0.7° = ~78 км), 19 переменных (t2m, u10, v10, msl, z500, t850, ...), obs_window=2 (два временых шага входа, +6h каждый).

Mesh: икосаэдр levels [4, 6], ~40,962 вершины.

Output: полное состояние атмосферы через +6h, c авторегрессионным rollout до +24h (4 шага).

Модель: ~5.9M параметров, InteractionNet с edge features и residual connections.

## Наша цель

**Улучшить качество прогноза для конкретного региона** (Красноярский край: 50–60°N, 83–98°E) без существенной деградации глобального прогноза.

## Что уже сделано

### 1. Глобальная модель (baseline)
- Grid 512×256 (~0.7°), 19 переменных
- Skill = 57.3% (mean over all variables), ACC = 0.988/0.956 (+6h/+24h)
- В ROI Красноярска: ~294 grid-точки (14 lat × 21 lon)

### 2. Multires fine-tuning (текущий лучший результат)

**Идея**: заменить грубые grid-ноды в ROI на мелкую сетку 0.25° и дообучить модель.

**Как это работает**:
1. Берём глобальный grid 512×256
2. **Вырезаем** 294 грубые точки в ROI
3. **Вставляем** 2,501 мелких точек (0.25°, сетка 41×61) в ROI
4. Получается flat grid из 133,279 нод (нерегулярный — грубый снаружи, мелкий внутри)
5. Строим GNN-графы (grid↔mesh) для этого нового грида
6. Fine-tuning: грузим pretrained веса глобальной модели, дообучаем с ROI-only loss

**Стратегия обучения**:
- Freeze processor на 6 эпох → encoder/decoder адаптируются к новой геометрии
- Unfreeze на эпоху 7 с пониженным lr для processor
- Curriculum: AR=1 → AR=2 → AR=3 → AR=4 (по 8 эпох на каждый уровень)

**Результат**: Skill = **75.8%** (ROI), t2m RMSE = 0.96°C (+6h), 1.40°C (+24h)

### 3. DualMesh (провал)

Пробовали добавить **отдельный региональный GNN-модуль** (~1M параметров) поверх замороженной глобальной модели. Модуль получал detached латенты глобальной модели и пытался вычислить поправку. **Не сработало**: correction ≈ 0, Skill = 71.3% (хуже чем multires).

## КЛЮЧЕВАЯ ПРОБЛЕМА (то, над чем нужна помощь)

### Данные для обучения — интерполированные, не настоящие

Обучающие данные для 2,501 мелких точек в ROI получены **билинейной интерполяцией** из грубого глобального грида (~0.7°). То есть:
- Мелкая сетка 0.25° содержит **ровно ту же информацию**, что и грубая сетка 0.7°
- Нет реального мелкомасштабного сигнала в training data
- Модель по сути учится делать **гладкую интерполяцию** — никаких субпиксельных деталей

При этом **реальные данные ERA5 0.25° для региона существуют** (скачаны с CDS API, 2010–2020, те же 19 переменных, на сетке 61×41), но они **не использовались при обучении** — только для тестирования.

### Суть проблемы — это "апскейл"

По сути мы делаем **super-resolution**: mesh (~170 вершин в ROI) декодирует в 2,501 мелких точек (вместо прежних 294 грубых). Каждая mesh-нода теперь отвечает за ~15 grid-точек вместо ~2.

Decoder (`CONTAINED` mode): каждая grid-точка получает взвешенную сумму от 3 mesh-вершин (из треугольника, в который попадает точка). Для мелких точек, стоящих близко друг к другу, эти 3 mesh-вершины часто одни и те же → соседние grid-точки получают почти одинаковый прогноз.

## Что нам нужно

Предложи **конкретные архитектурные и методологические идеи**, как улучшить региональный прогноз. Нас интересуют подходы по нескольким направлениям:

### Направление 1: Как использовать реальные 0.25° данные для обучения?

У нас есть:
- Глобальные данные 0.7° для всего мира (2010–2021)
- Региональные данные 0.25° для Красноярска (2010–2020)

Как правильно их совместить для обучения? Наивный merge создаёт разрыв на границе ROI (разная физика на разных разрешениях). Нужна стратегия:
- Как обрабатывать граничные эффекты?
- Нужен ли consistency loss между грубым и мелким прогнозом?
- Как нормализовать данные (scaler от глобальных или региональных)?

### Направление 2: Архитектурные улучшения decoder'а

Сейчас decoder — это 3 слоя GCN, одинаковых для всех точек. Для ROI с 2,501 точкой, декодируемой из ~170 mesh-вершин, нужна лучшая стратегия. Идеи:
- Иерархический decoder (mesh → coarse grid → fine grid)
- Attention-based decoding для ROI
- Skip connections из encoder'а (U-Net style)
- Learned upsampling kernel

### Направление 3: Изменение mesh-структуры

Сейчас mesh одинаковый везде (icosahedral level 4–6). Можно ли:
- Добавить level 7/8 mesh **только в ROI**?
- Как правильно соединить multi-resolution mesh (рёбра между уровнями)?
- Не приведёт ли это к oversmoothing из-за разной density?

### Направление 4: Альтернативные подходы

- Каскад: глобальная GNN → региональный UNet/CNN для super-resolution
- Nested mesh: обрабатывать ROI на отдельном мелком mesh, с boundary conditions от глобальной модели
- Learned interpolation: обучить отдельный модуль, который берёт грубый прогноз + рельеф/маску земли и делает downscaling

## Важные ограничения

1. **Вычислительные ресурсы**: 1× NVIDIA GPU (T4/A100), обучение ~2ч на эпоху для 133K-нодного графа
2. **Данные**: 2010–2021 (6-часовой шаг), ~8000 сэмплов train, ~900 val
3. **Модель уже pretrained**: глобальная модель обучена, хотим fine-tune, а не from scratch
4. **Framework**: PyTorch + torch_geometric
5. **Метрики**: Skill = 1 - RMSE_model/RMSE_climatology, ACC (spatial anomaly correlation)
